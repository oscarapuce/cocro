## 2. Bug Fixes (Critical)

### 2.1 🔴 Cache re-activation on INTERRUPTED → PLAYING transition

**Problem :** When `HeartbeatTimeoutScheduler` or `LeaveSessionUseCase` interrupts a session, they call `sessionGridStateCache.deactivate(sessionId)` which removes the session from `sessions:active`. When a user later joins and the session transitions back to PLAYING, `JoinSessionUseCase` does **not** re-add the session to the active set.

**Impact :**
- `SessionFlushScheduler` won't flush this session's grid state → data loss risk
- `HeartbeatTimeoutScheduler` won't monitor this session → zombie participants

The issue self-heals on the first `UpdateSessionGridUseCases` call (which does a cache-miss fallback `initialize()`), but there's a gap between rejoin and first grid update where the session is untracked.

**Fix :**

```
File: JoinSessionUseCase.kt
```

After `sessionRepository.save(updatedSession)`, detect the INTERRUPTED → PLAYING transition:

```kotlin
// Re-activate Redis cache when session resumes from INTERRUPTED
if (session.status == SessionStatus.INTERRUPTED && updatedSession.status == SessionStatus.PLAYING) {
    val gridState = sessionGridStateCache.get(session.id) ?: session.sessionGridState
    sessionGridStateCache.initialize(session.id, gridState)
    logger.info("Session {} cache re-activated (INTERRUPTED → PLAYING)", session.shareCode.value)
}
```

**Files :** `JoinSessionUseCase.kt`  
**Tests :** Add test case in `JoinSessionUseCaseTest.kt`  
**Effort :** 1h  
**Dependencies :** None — do first

### 2.2 🔴 Docs say INTERRUPTED join is restricted — code says otherwise

**Problem :** `lifecycle.md` transition table says INTERRUPTED sessions can only be joined by "previously LEFT participant". The actual code in `Session.applyJoin()` allows **any authenticated user** to join INTERRUPTED sessions — the guard is only `status in [PLAYING, INTERRUPTED]`, with no prior-participant check.

**Fix :** Documentation update only — no code change needed.

```
Files: docs/sessions/lifecycle.md, docs/sessions/session-ddd.md
```

Update transition table:
```
| INTERRUPTED | Join | any authenticated user | PLAYING | ParticipantJoined broadcast |
```

Update session-ddd.md §2 Join rules:
> INTERRUPTED sessions can be joined by any authenticated user (new or previously LEFT). The session transitions back to PLAYING on the first join.

**Effort :** 30min  
**Dependencies :** None

### 3.2 🟢 INTERRUPTED session expiry scheduler

**Problem :** INTERRUPTED sessions accumulate forever in MongoDB.

#### BFF

| File | Change |
|------|--------|
| `SessionRepository.kt` (port) | Add `findInterruptedBefore(cutoff: Instant): List<Session>` |
| `MongoSessionRepositoryAdapter.kt` | Implement: `status = INTERRUPTED AND updatedAt < cutoff` |
| `InterruptedSessionExpiryScheduler.kt` | **New** in `infrastructure/scheduler/` |
| `application.yml` | Add `cocro.session.interrupted.expiry-days: 7` |

Scheduler logic:
```kotlin
@Scheduled(cron = "0 0 3 * * *")  // daily at 3 AM
fun expireInterruptedSessions() {
    val cutoff = Instant.now().minus(Duration.ofDays(expiryDays))
    val expired = sessionRepository.findInterruptedBefore(cutoff)
    for (session in expired) {
        session.participants.forEach { p -> heartbeatTracker.remove(session.id, p.userId) }
        sessionGridStateCache.deactivate(session.id)
        sessionRepository.deleteById(session.id)
        logger.info("Expired INTERRUPTED session {} (last updated {})", session.shareCode.value, session.updatedAt)
    }
}
```

**Decision point :** Hard delete vs soft delete (add `EXPIRED` status). Recommend hard delete for MVP — `EXPIRED` status adds query complexity for no user-facing benefit yet. Revisit for v0.3.0 "Historique des parties".

**Effort :** 3h  
**Dependencies :** None, but do after bug fix 2.1

## 4. Robustness / WebSocket

### 4.1 🔵 WebSocket retry with backoff on SyncRequired

**Problem :** `resync()` in `grid-player.component.ts` calls `syncSession.execute()` once with no retry and **no error handler**. Network failure silently leaves the client stale.

#### Angular

| File | Change |
|------|--------|
| `grid-player.component.ts` | Wrap `syncSession.execute()` with RxJS `retry` + exponential backoff |

```typescript
private resync(_targetRevision: number): void {
  this.letterAuthors.clearAll();
  this.syncing.set(true);
  this.syncSession.execute(this.shareCode()).pipe(
    retry({ count: 3, delay: (_err, attempt) => timer(1000 * Math.pow(2, attempt - 1)) }),
  ).subscribe({
    next: (full: SessionFullResponse) => {
      this.revision.set(full.gridRevision);
      this.participantCount.set(full.participantCount);
      this.participants.set(full.participants ?? []);
      this.status.set(full.status);
      this.selector.clearAllLetters();
      full.cells.forEach((c: CellStateDto) => {
        if (c.letter) this.selector.setLetterAt(c.x, c.y, c.letter);
      });
      this.syncing.set(false);
    },
    error: () => {
      this.error.set('Synchronisation échouée. Rechargez la page.');
      this.syncing.set(false);
    },
  });
}
```

Add `syncing = signal(false)` and optionally disable grid input while syncing.

**BFF :** No changes.

**Effort :** 2h  
**Dependencies :** None

### 4.2 🔵 Gap detection (revision gap triggers auto-sync)

**Problem :** If the client misses a `GridUpdated` broadcast, its local revision drifts. Currently `GridUpdated` doesn't carry a `revision` field, so the client can't detect gaps.

#### BFF

| File | Change |
|------|--------|
| `SessionEvent.kt` | Add `val revision: Long` to `GridUpdated` data class |
| `UpdateSessionGridUseCases.kt` | Pass `newState.revision.value` when constructing `GridUpdated` event |

```kotlin
SessionEvent.GridUpdated(
    actorId = user.userId(),
    posX = dto.posX,
    posY = dto.posY,
    commandType = dto.commandType,
    letter = dto.letter,
    revision = newState.revision.value,  // ← NEW
)
```

#### Angular

| File | Change |
|------|--------|
| `session-events.model.ts` | Add `revision?: number` to `GridUpdatedEvent` |
| `grid-player.component.ts` | Update `onGridUpdated()` |

```typescript
private onGridUpdated(event: GridUpdatedEvent): void {
  if (event.actorId === this.myUserId()) return;

  // Gap detection: if revision jumps, trigger full resync
  if (event.revision && event.revision > this.revision() + 1) {
    this.resync(event.revision);
    return;
  }

  this.revision.set(event.revision ?? this.revision() + 1);
  // ... apply cell update as before
}
```

**Effort :** 3h  
**Dependencies :** ITEM 4.1 (retry backoff) — gap detection triggers resync
