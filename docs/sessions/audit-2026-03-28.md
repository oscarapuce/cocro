  │                                                   │── POST /api/sessions/join { shareCode }
  │   ┌──────────────────────────────────┐            │◄─ 200 SessionFullDto

  │   │ ngOnInit():                      │            │
  │   │  1. joinSession.execute()        │            │

### Flux de déconnexion / heartbeat

```
Client connecté (ACTIVE)
  │
  │── perte WS / navigation ──► StompSessionEventListener.onDisconnect()
  │                                 └── heartbeatTracker.markAway(sessionId, userId)
  │                                        ┌──────────────────────────────┐
  │                                        │ AWAY (grace period = 30s)   │
  │                                        └──────────────────────────────┘
  │                                                   │
  │   ┌── reconnexion < 30s ──┐       ┌── timeout > 30s ──┐
  │   │ POST /join → isAway   │       │ HeartbeatTimeout-  │
  │   │ → markActive()        │       │ Scheduler checks   │
  │   │ → transparent (pas de │       │ → session.leave()  │
  │   │   broadcast)           │       │ → ParticipantLeft  │
  │   └────────────────────────┘       │   (reason:timeout) │
  │                                    │ → si dernier:      │
  │                                    │   INTERRUPTED      │
  │                                    └────────────────────┘
```

## 3. Incohérences identifiées

### 🔴 INC-1 — Heartbeat WS endpoint absent côté BFF + Angular

**Gravité : CRITIQUE**

La documentation (`lifecycle.md`, `session-flow.html`, `cache.md`) mentionne un endpoint STOMP :
```
SEND /app/session/{code}/heartbeat
```
que le client doit envoyer toutes les ≤30s pour rester ACTIVE.

**Réalité dans le code :**
- **BFF** : `SessionWebSocketController` n'a **aucun** `@MessageMapping("/session/{shareCode}/heartbeat")`. Le seul handler est `/session/{shareCode}/grid`.
- **Angular** : `SessionStompAdapter` et `GridPlayerComponent` n'envoient **aucun** heartbeat périodique. Pas de `setInterval`, pas de `heartbeat` dans le code Angular.

**Conséquence** : Le système repose uniquement sur le STOMP disconnect event (`StompSessionEventListener`) pour détecter les déconnexions. Un utilisateur qui garde son onglet ouvert mais dont le réseau est dégradé (sans que le WS se ferme) **ne sera jamais marqué away**, ni timeout. Le STOMP-level heartbeat natif (heart-beat header) maintient la connexion WS mais ne nourrit pas le `HeartbeatTracker` de l'application.

### 🔴 INC-2 — `SessionConnectEventListener` : code mort (Welcome dupliqué)

**Gravité : MOYENNE**

Deux mécanismes envoient un `SessionWelcome` :
1. `SessionConnectEventListener` : écoute `SessionConnectedEvent` → envoie `SessionWelcome` sur `/user/queue/session` via `notifyUser()`
2. `SessionWebSocketController.onWelcomeSubscribe()` : `@SubscribeMapping("/session/{code}/welcome")` → retourne `SessionWelcome` synchrone

Le test d'intégration (`SessionWebSocketIT`) utilise exclusivement le pattern `@SubscribeMapping`. L'Angular (`SessionStompAdapter`) utilise aussi exclusivement `subscribeWelcome()` (pattern 2).

**Conséquence** : `SessionConnectEventListener` est du code mort. Un client recevra **deux** `SessionWelcome` s'il s'abonne aussi à `/user/queue/session` (ce que fait l'Angular dans `subscribeTopic` → `privateSub`).

Cas concret : L'Angular subscribe à `/user/queue/session` dans `subscribeTopic()` (pour les `SyncRequired`). Si le timing est bon, il pourrait recevoir le Welcome envoyé par `SessionConnectEventListener` via `/user/queue/session`, parsé comme `SyncRequiredEvent` (type mismatch) → erreur silencieuse ou comportement inattendu.

### 🟡 INC-3 — Type assertion incomplète dans `subscribeTopic()`

**Gravité : FAIBLE** (TypeScript seulement, pas de bug runtime)

Dans `session-stomp.adapter.ts` :
```typescript
const event = JSON.parse(msg.body) as
  | ParticipantJoinedEvent
  | ParticipantLeftEvent
  | GridUpdatedEvent
  | GridCheckedEvent;
```

**Manquent** : `SessionEndedEvent` et `SessionInterruptedEvent`. Ces événements arrivent bien via `/topic/session/{code}` mais ne sont pas dans le type union. Le cast TypeScript ne filtre pas au runtime, donc les events arrivent mais sans typage correct — le `handleEvent()` dans `GridPlayerComponent` les traite correctement via `switch(event.type)`.

### 🟡 INC-4 — `GridCheckResponse` vs `GridCheckSuccess` : champs manquants

**Gravité : FAIBLE**

Angular `GridCheckResponse` :
```typescript
{ isComplete, isCorrect, correctCount, totalCount }
```

BFF `GridCheckSuccess` (réponse REST) :
```kotlin
{ shareCode, isComplete, isCorrect, filledCount, totalCount, wrongCount, correctCount }
```

Angular ignore `shareCode`, `filledCount`, `wrongCount`. Ce n'est pas un bug mais les données supplémentaires sont disponibles si l'UX en a besoin.

### 🟡 INC-5 — `GridChecked` broadcast event : pas de `isCorrect`

**Gravité : MOYENNE**

Le BFF `SessionEvent.GridChecked` **ne contient pas `isCorrect`** :
```kotlin
data class GridChecked(
    val userId: String,
    val isComplete: Boolean,
    val correctCount: Int,
    val totalCount: Int,
) : SessionEvent
```

L'Angular `GridCheckedEvent` **ne contient pas `isCorrect`** non plus. Cependant, le document `session-flow.html` (§ Events) liste `GridChecked` avec `{ isComplete, isCorrect, correctCount, totalCount }` — c'est faux, `isCorrect` n'est pas dans l'event broadcast.

Le `GridPlayerComponent.checkGrid()` contourne le problème en utilisant la réponse HTTP (qui a `isCorrect`) plutôt que l'event broadcast. Mais les **autres participants** qui reçoivent le broadcast `GridChecked` n'ont pas accès à `isCorrect`.

### 🟡 INC-6 — `ngOnDestroy` ne fait pas `POST /leave`

**Gravité : MOYENNE**

Dans `GridPlayerComponent.ngOnDestroy()` :
```typescript
ngOnDestroy(): void {
    this.sessionSocket.disconnect();
    this.selector.initGrid(createEmptyGrid('0', '', 10, 10));
    this.letterAuthors.clearAll();
}
```

Seul `disconnect()` est appelé (fermeture STOMP). **Pas de `leaveSession.execute()`**. Le serveur détectera la déconnexion STOMP et lancera le flow heartbeat/timeout (30s grace → eviction), mais :
- Si l'utilisateur navigue ailleurs dans l'app (pas de fermeture navigateur), la session STOMP est fermée mais pas de leave explicite → 30s de latence avant que les autres voient le départ.
- Le bouton "Quitter" (`leave()`) fait correctement le `POST /leave` + `disconnect()`.
- Navigation par le routeur Angular (back, changement d'URL) → `ngOnDestroy` → pas de leave.

### 🟡 INC-7 — Pas de `beforeunload` / `canDeactivate` guard

**Gravité : MOYENNE**

Aucun `@HostListener('window:beforeunload')` ni `canDeactivate` guard sur la route `/play/:shareCode`. Si l'utilisateur ferme l'onglet ou navigue vers un site externe :
- Le WS se ferme → `StompSessionEventListener` → `markAway` → grace period 30s → timeout.
- Pas de leave propre.

### 🟡 INC-8 — Pas de `StartSession` côté Angular

**Gravité : FAIBLE** (pas dans le flow actuel)

Le BFF a un endpoint `POST /api/sessions/start` documenté dans `security.md` mais aucun use case correspondant n'est visible dans le code actuel. Le state machine n'a plus de transition `start` (anciennement CREATING→PLAYING, supprimé). Vestige possible dans la config de sécurité.

### 🟡 INC-9 — `SyncRequired` sur `/user/queue/session` : race avec Welcome du `SessionConnectEventListener`

**Gravité : MOYENNE** (lié à INC-2)

L'Angular subscribe à `/user/queue/session` pour recevoir `SyncRequired`. Mais `SessionConnectEventListener` envoie aussi `SessionWelcome` sur ce même canal. Si le client est déjà connecté et subscribed à `/user/queue/session` quand le `SessionWelcome` arrive, il sera parsé dans le handler `privateSub` qui type-cast en `SyncRequiredEvent` → potentiel crash ou données mal interprétées.

## 5. Angles morts et risques

### 🔴 RISK-1 — Heartbeat applicatif absent = timeout impossible sans déco WS

Sans le heartbeat endpoint, le seul signal de vie est la connexion STOMP. Un client avec un réseau dégradé (connexion half-open, onglet en arrière-plan avec le WS maintenu) ne sera **jamais** marqué away. Le `HeartbeatTracker` est alimenté uniquement par :
- `markActive()` : appelé dans `JoinSessionUseCase` (au join/rejoin)
- `markAway()` : appelé dans `StompSessionEventListener` (au disconnect STOMP)

Pas d'alimentation périodique → le tracker est binaire : "joined" ou "disconnected".

### 🟡 RISK-2 — Sessions INTERRUPTED immortelles

Pas de TTL, pas de scheduler de nettoyage pour les sessions INTERRUPTED/ENDED en MongoDB/Redis. Les clés Redis `session:{id}:*` persistent indéfiniment. Documenté dans `session-flow.html` blindspot 2.

### 🟡 RISK-3 — Pas de retry/rattrapage pour `SyncRequired`

Si le message privé `SyncRequired` est manqué (onglet en arrière-plan, micro-coupure STOMP), le client diverge silencieusement. Pas de mécanisme de heartbeat ou de polling de revision pour détecter le drift.

### 🟡 RISK-4 — `SessionConnectEventListener` double Welcome

Le `SessionConnectEventListener` écoute `SessionConnectedEvent` et envoie un `SessionWelcome` sur `/user/queue/session`. Ce code est probablement vestigial (le pattern `@SubscribeMapping` le remplace). Mais il est actif et produit un message sur un canal que l'Angular écoute pour les `SyncRequired`.
