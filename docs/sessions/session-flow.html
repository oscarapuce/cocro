⚡ CAS Conflict (grid update)

Client envoie une mise à jour grid
compareAndSet() détecte un conflit de révision
BFF récupère la révision courante (Redis → MongoDB fallback)
Envoie SyncRequired { currentRevision } (privé)
Retourne 409 ConcurrentModification
Client appelle POST /sync → réhydrate la grille
Client reprend l'édition depuis la bonne révision

⚠️ Blindspots & Gaps potentiels

1. INTERRUPTED — nouveaux participants bloqués

Seuls les participants previously-LEFT peuvent rejoindre une session INTERRUPTED. Un nouvel utilisateur qui n'a jamais rejoint ne peut pas entrer. Comportement non documenté côté client.

2. Sessions immortelles

Les sessions INTERRUPTED persistent indéfiniment en MongoDB + Redis. Pas de TTL, pas de job de nettoyage. Une session abandonnée depuis des mois reste en base.

3. POST /sync → participant JOINED requis

Si un user reconnecte (STOMP CONNECT) mais n'a pas encore appelé POST /join, il ne peut pas appeler POST /sync. La séquence CONNECT → /join → sync doit être strictement respectée.

4. Créateur doit appeler POST /join séparément

POST /api/sessions crée la session mais n'ajoute pas le créateur comme participant. Il doit immédiatement appeler POST /join — un pattern UX à deux appels peu intuitif.

5. SyncRequired sans retry

SyncRequired est envoyé une seule fois en message STOMP privé. Si le client le rate (onglet caché, micro-déconnexion), pas de mécanisme de rattrapage — il dérive silencieusement.

6. GridEditor (Phase 4) absent

La création de grilles n'est pas encore implémentée dans cocro-web. Seules les grilles insérées via API/DB fonctionnent. Le flux complet de bout en bout est incomplet.
